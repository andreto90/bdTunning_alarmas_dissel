Optimización controlada del motor Oracle 11g Enterprise Edition

ERP Yeminus | Diagnóstico, ajuste de memoria, validación postcambio y evidencia gráfica

Sistema evaluadoERP Yeminus
MotorOracle Database 11g Enterprise Edition Release 11.2.0
Servidor32 GB RAM | 8 CPU cores
Fecha del informe2026-03-06

Este documento consolida la línea base técnica, las actividades ejecutadas, las mediciones obtenidas y la evaluación posterior a la intervención realizada sobre el motor Oracle 11g que soporta el ERP Yeminus. El alcance se limitó deliberadamente a ajustes seguros sobre el motor y validación de estado, sin alterar sentencias SQL ni índices del ERP para evitar riesgos sobre el comportamiento funcional del sistema.

1 Resumen ejecutivo

Se ejecutó una optimización y afinamiento controlado del motor Oracle 11g Enterprise enfocado en parámetros de memoria, reutilización de cursores y validación de estado general del host y la instancia. El análisis inicial mostró una configuración con AMM limitada a 13.75 GB sobre un servidor con 32 GB de RAM, un buffer cache hit ratio base de 89.73%, baja presión de CPU, baja utilización de sesiones/procesos y ausencia de contención severa. Bajo ese contexto se incrementó la memoria total administrada por Oracle a 20 GB y se elevó session_cached_cursors de 50 a 100, manteniendo intactas las estructuras y consultas del ERP Yeminus.

No se modificaron índices, planes de ejecución ni consultas del ERP, por tratarse de componentes de aplicación bajo control funcional del sistema Yeminus.

El motor quedó operando con memory_target = 20 GB y memory_max_target = 20 GB, manteniendo Automatic Memory Management.

La evidencia posterior al reinicio muestra una SGA máxima de 20.39 GB, Buffer Cache inicial de 6.27 GB, Shared Pool de 1.28 GB y 12.6 GB de memoria aún disponibles para crecimiento dinámico según carga.

El buffer cache hit ratio medido inmediatamente después del reinicio cayó a 83.61%; esto es esperable por caché fría y no se interpreta como deterioro estructural.

El host permanece sin presión de CPU y sin saturación de processes/sessions; por tanto, el beneficio esperado del ajuste es preventivo y de mejor aprovechamiento de memoria, no una corrección de sobrecarga crítica.

2 Alcance, restricciones y criterio de intervención

El trabajo se diseñó bajo un principio de riesgo mínimo. Se ejecutaron únicamente actividades que no comprometen datos ni estructura funcional del ERP. El alcance incluyó diagnóstico de parámetros, eventos de espera, SQL costosos, segmentación de crecimiento, objetos inválidos, consumo de memoria, utilización de CPU y límites de sesiones/procesos.

4.7 Consumo de almacenamiento por estadísticas y logs del motor

Durante la auditoría de crecimiento, se evidenció que la base de datos acumulaba un alto volumen de información histórica correspondiente al repositorio de carga de trabajo (AWR) y a estadísticas del optimizador. Esta data residual generaba un engrosamiento progresivo del tablespace SYSAUX, afectando los tiempos de respuesta al consultar el diccionario de datos (overhead administrativo) e impactando el almacenamiento físico sin aportar valor directo a la operación del ERP Yeminus.

Maceramiento y almacenamiento: El crecimiento descontrolado de estos componentes satura el subsistema de discos. 6.3 Interpretación del buffer cache hit ratio posterior

El buffer cache hit ratio medido inmediatamente después del reinicio fue 83.61%, inferior al 89.73% observado antes del cambio. Técnicamente esto no representa un deterioro estructural, sino el efecto esperado de un buffer cache vacío después del arranque: la instancia todavía no había “calentado” la memoria con bloques de datos frecuentemente utilizados.

Conclusión operativa: el ratio debe re-medirse con la instancia estabilizada y con carga habitual de negocio. La expectativa razonable no es evaluarlo en la ventana inmediata postinicio, sino luego de 24 a 72 horas de operación real.

6.4 Presión real de PGA y validación de CPU posterior

La revisión de PGA mostró una utilización muy baja respecto de la capacidad disponible: total PGA allocated de 113.87 MB, total PGA inuse de 77.09 MB, maximum PGA allocated de 116.59 MB, over allocation count = 0 y extra bytes read/written = 0. Este conjunto de métricas descarta presión de memoria de proceso y confirma que el problema no estaba en PGA insuficiente.

8 Riesgos controlados y decisiones de diseño adoptadas

  • No se ejecutaron rebuilds de índices ni cambios sobre SQL del ERP para evitar desalinear el soporte funcional de Yeminus.
  • No se fijaron manualmente db_cache_size o shared_pool_size porque la instancia quedó correctamente bajo AMM con 20 GB; forzar pools manuales habría sido contradictorio con el objetivo de redistribución automática.
  • La evaluación postreinicio se interpretó con criterio técnico, reconociendo el sesgo natural de un buffer cache recién reiniciado.
  • Las métricas de PGA y CPU descartaron ajustes adicionales de procesador o de memoria de proceso.

9 Recomendaciones técnicas posteriores

Re-medición operativa del buffer cache hit ratio después de 24–72 horas de carga normal, para verificar el comportamiento con caché caliente.

Monitoreo de v$sgainfo y v$memory_dynamic_components para observar el crecimiento automático de Buffer Cache y Shared Pool según patrón de uso real.

Revisión funcional conjunta con proveedor/ERP sobre SQL repetitivos que consultan metadatos de all_constraints/all_cons_columns, dado que constituyen parte importante del tiempo acumulado.

Revisión de los 40 objetos inválidos detectados, priorizando vistas con ORA-00942 / ORA-04063, por su potencial impacto sobre procesos de negocio o reportes.

Seguimiento del crecimiento de segmentos voluminosos, especialmente LOBs y tablas MTODETALLADO / MTODETALLADO_INV / AU_LOGS, para anticipar necesidades de tablespace y de cache.

Mantener session_cached_cursors en 100 y la política de memoria en 20 GB salvo evidencia futura de comportamiento anómalo.

10 Conclusión final

La intervención realizada sobre Oracle 11g fue técnicamente apropiada, segura y consistente con las restricciones operativas del entorno Yeminus. El cambio principal consistió en elevar la memoria total administrada por Oracle desde 13.75 GB hasta 20 GB, junto con el aumento de session_cached_cursors de 50 a 100. El estado posterior confirma que la instancia quedó estable, con holgura de CPU, baja ocupación de procesos/sesiones, PGA sin presión y capacidad para que Oracle expanda dinámicamente el Buffer Cache conforme se estabilice la carga.

Dado que no se tocaron índices ni consultas del ERP, el informe concluye que el trabajo efectuado optimizó correctamente el motor dentro del margen de seguridad permitido. La principal mejora esperada es un mejor aprovechamiento de memoria y una reducción gradual de lecturas físicas una vez el cache se caliente; la optimización adicional más relevante en el futuro dependerá de decisiones sobre SQL y objetos funcionales del ERP, no de nuevos ajustes agresivos del motor.

SQL*Plus - Buffer cache hit ratio inmediato postreinicio
SQL*Plus: Release 11.2.0.4.0 Production
Connected to: Oracle Database 11g Enterprise Edition Release 11.2.0
SQL> SELECT ROUND((1 - (phy.value / (cur.value + con.value))) * 100, 2) / 100 AS BUFFER_CACHE_HIT_RATIO FROM v$sysstat phy, v$sysstat cur, v$sysstat con WHERE phy.name = 'physical reads' AND cur.name = 'db block gets' AND con.name = 'consistent gets';
BUFFER_CACHE_HIT_RATIO
----------------------
0.8361
